The Shipyard

design → build → release for a fleet of agent-maintained repos

Sixteen personal projects run under autonomous maintenance agents on systemd timers — tests re-run nightly, feedback triaged into PRs, failures triaged on-call, docs refreshed. It sounds like a shipping pipeline. Then I measured it.

16
repos under agent maintenance (8 public · 8 private)
871

0

8.8%

871 runs, zero catches. The release agent was re-running the same tests the authoring session had already made green — validating yesterday’s work with the same signal the author optimized against. And design wasn’t a loop at all: every feature started as a chat conversation, which means it started whenever I happened to notice something, not when the data did.

This deck is the design record for rebuilding that system as Shipyard — borrowing the three-loop framing from agent-ready-repo: supervised loops with human gates at every irreversible handoff.

Three loops, five crew

Design, build, and release are peer loops. Two agents sit outside the shipment paradigm entirely: an on-call orchestrator and a historian. Nothing crosses a gate without a human stamp.

Skills-parity. The agents run the same skill files a human-driven session invokes: polish-ticket hardens a spec until a zero-context agent can build it safely; execute-ticket orchestrates the build with mandatory on-system verification. Mentat calls one, helldiver calls the other, I call both by hand. One implementation, three callers — no agent-only forks of the workflow.

Installable, gated, judged

Shipyard installs per-project, like a package. Names are a knob, not a doctrine — canonical role IDs (design, build, release, medic, scribe) live in code and events; the display names come from a theme:

bash install.sh --project <dir> --theme spacetime   # mentat · helldiver · proctor · medic · chronicler
bash install.sh --project <dir> --theme plain       # design · build · release · medic · scribe

The hardening that comes with it: merge is opt-in (default no), a repo with zero CI checks reads as red, the merge→validate→revert loop is tested against real git topologies, and every agent behavior sits behind a test that was first shown failing.

The 5-day trial

After install across the fleet, the system — not any one agent — is judged over five days, each criterion with a floor and an evidence trail:

CriterionFloor
Bugs caught & fixed≥1 real defect caught by release/on-call and fixed through the loop (baseline: 0 in 871 runs)
Usage assessedreal-telemetry usage assessments for ≥3 projects in the daily dispatch
Features shipped≥1 feature end-to-end: proposal → stamp → build → release → merged
Decisions surfaced≥1 consequential choice surfaced to the human and decided, not defaulted

Any floor missed becomes a written finding with a follow-up ticket — not a silent pass.
